<h2 id="marks">Core inline marks</h2>

Marks are inline formatting carried on runs of text. The CRDT names are `strong`/`emphasis`/`strike` (not bold/italic/strikethrough). The togglable ones flip across the selection with `change((c) => c.setMark(name))`; `link` carries data, so it is applied with an explicit range and attrs — `setMark(name, { attrs, range })` — instead.

<h2 id="math">Optional math</h2>

Math is a feature, not a core node type. The root `@tasfer/editor` entry and
`baseSchema` do not load `@tasfer/tex`, so hosts that do not need formulae
pay no math dependency or authoring cost. Install display math (`$$…$$`) and
inline math (`$…$`) together from the dedicated entry point:

```bash
npm install @tasfer/editor @tasfer/tex
```

```ts
// appDataSchema.ts — safe to import from workers and persistence code
import { baseDataSchema } from "@tasfer/editor";
import { mathDataExtension } from "@tasfer/editor/math/data";

export const appDataSchema = baseDataSchema.extend(mathDataExtension());

// editorSchema.ts — imported only by the interactive editor surface
import { baseSchema, createEditor } from "@tasfer/editor";
import { mathExtension } from "@tasfer/editor/math";

export const appSchema = baseSchema.use(mathExtension());

const editor = createEditor({
  element,
  markdown: String.raw`Euler: $e^{i\pi}+1=0$`,
  schema: appSchema,
});
```

`appSchema` is the canvas/editor half; the independently composed
`appDataSchema` is the worker-safe persistence and serialization half. Each
piece registers once, on the spec that owns it: the math block spec carries its
codec and the `$$…$$` display recognizer, the math mark spec carries the `$…$`
inline recognizer and the structured-mark behavior, and a `structuredKinds`
entry for the shared `"math"` kind carries the snapshot clone adapter. The full
`appSchema` additionally installs the rendering classes, the `"math"` kind's
selection adapters (the clipboard serializer and the range resolver that keeps
committed selections construct-atomic), and the live input rules. Keep the two
definitions paired, but keep their modules separate so a worker importing the
data schema cannot evaluate the interactive graph.

If a worker should not import the interactive math view, compose the data-only
entry there. It registers the specs above — codecs, Markdown syntax,
structured-mark behavior, and the kind's clone adapter — without constructing
`MathNode`, `MathMark`, or the live input/selection stack (the kind's selection
adapters resolve selections through the `@tasfer/tex` layout engine, so they
deliberately belong to the full extension):

```ts
import { baseDataSchema } from "@tasfer/editor";
import { mathDataExtension } from "@tasfer/editor/math/data";

export const workerDataSchema = baseDataSchema.extend(mathDataExtension());
```

Structured `content_edit` operations are schema-agnostic and replay through
this data schema. Interactive editors install the full extension:

```ts
const mathSchema = baseSchema.use(mathExtension());
```

Display and inline editing both use structured `MathDocument` operations.
Every creation path — markdown import, the `$…$`/`$$` input rules, wrapping a
selection, block conversion — creates the structured document immediately; the
tree is the only source of a formula's content.

```ts
import {
  createDoc,
  createEditor,
  parsePage,
  serializeToMarkdown,
  tokenizePage,
} from "@tasfer/editor";

const doc = createDoc({ markdown: savedMarkdown, schema: appDataSchema });
const attachedEditor = createEditor({ element, doc, schema: appSchema });
const page = parsePage(
  tokenizePage(importedMarkdown, appDataSchema),
  appDataSchema,
);
const output = serializeToMarkdown(page.blocks, undefined, {
  schema: appDataSchema,
});
```

Without `mathExtension`, dollar-delimited input remains literal text: the
parser preserves both `$...$` and `$$...$$` source instead of silently
interpreting or discarding it. That lets a math-free client retain content from
a newer or differently configured peer until the feature is installed. In a
math-enabled document an inline equation stores no LaTeX in block characters
at all: the mark anchors on a single object-replacement placeholder character
(U+FFFC) and its content lives wholly in the referenced `MathDocument`
attachment. A display block's flat text is empty; its content is the
block-authority document. A host without the math feature renders the
placeholder glyph.
The feature also exposes `MathDocument`,
`mathDocumentToStructured`, `structuredToMathDocument`, and
`parseMathDocumentInit`; editor core supplies the generic `content_edit`
tree/text CRDT those adapters target. Tree-backed display math owns its block;
tree-backed inline math is a supplemental attachment referenced by the covering
mark. Neither path dual-writes a second editable LaTeX representation.
The two forms convert into each other losslessly: Backspace at a display
equation's leading edge demotes it to an inline chip, and converting a block
whose only content is one inline chip to `math` — `setBlock({ type: "math" })`,
a slash command, or a context menu — promotes the same tree to a display
equation.

<h2 id="extend">Extending</h2>

`use` installs a reusable feature bundle. Feature objects can contribute nodes,
marks, structured-kind adapters, live-input rules, action hooks, and theme or
string defaults. Markdown syntax recognizers and structured-content behavior
are not feature keys: they ride the node and mark specs themselves
(`markdownSyntax` on both, `structured` on marks), so each type registers once
with everything it owns. All facets are immutable and scoped to the resulting
schema, so two editors can install different feature sets on the same page:

```ts
import {
  baseSchema,
  defineMark,
  defineNode,
  type FeatureExtension,
} from "@tasfer/editor";

export const calloutFeature = {
  name: "callouts",
  nodes: [defineNode("callout", { attrs: { tone: { default: "note" } } })],
  marks: [defineMark("highlight", { attrs: { color: { default: "yellow" } } })],
} as const satisfies FeatureExtension;

const schema = baseSchema.use(calloutFeature);
```

Use `as const satisfies FeatureExtension` for exported feature values: it
checks the public feature contract while retaining literal node, mark, and
attribute types for the resulting `Editor` API. The optional `name` is metadata,
not a deduplication key; calling `use` twice composes twice, with the same
later-definition-wins behavior as `extend`.

For one-off registration, `extend` remains the lower-level API. It takes
_arrays_ of node and mark specs and returns a new, immutable schema:

```ts
const oneOffSchema = baseSchema.extend({
  nodes: [defineNode("one_off")],
});
```

Both methods leave the base untouched, so two editors on one page can run
different schemas. The schema's type is inferred and carried through `Doc`,
`Editor`, the React bindings, `ChangeApi`, and `query`; you do not pass a generic
manually:

```ts
const editor = createEditor({ element, schema });

editor.change((c) => {
  c.setBlock({ type: "callout", tone: "note" });
  c.setMark("highlight", { attrs: { color: "orange" } });

  c.setBlock({ type: "calluot" }); // TypeScript error: unknown block
  c.setMark("higlight"); // TypeScript error: unknown mark
});

const block = editor.query.block();
if (block?.type === "callout") {
  block.attrs.tone; // string
}
```

<h2 id="restrict">Restricting (allowed types)</h2>

`use` and `extend` decide which types _exist_. `restrict` decides which of them
the local user may _author_ — the ProseMirror-style allow-list. It returns a new
immutable schema, leaves the full registry (and therefore rendering) untouched,
and is the tool for constrained fields like a single-line title or a plain-text
comment box:

```ts
// A title field: the user can only ever produce a heading1, with no inline marks.
const titleSchema = baseSchema.restrict({ blocks: ["heading1"], marks: [] });
const title = createEditor({ element, schema: titleSchema });
```

- Omit a key to leave that dimension unrestricted; `marks: []` is a genuinely
  format-free field. Every name must already be registered (apply `use` /
  `extend` first, `restrict` last).
- The `paragraph` fallback is **always** creatable and cannot be excluded, so a
  document can never become unrepresentable — you never need to list it.
- The default `baseSchema` is unrestricted: without a `restrict` call every
  registered type is authorable, exactly as before.

**Authoring vs. rendering.** `restrict` gates _creation only_. Slash/menu
conversions, Enter-splits, Markdown shortcuts (`# `, `- `), paste, and
`change` API calls that would mint a disallowed type are refused (no-op) or
coerced to a plain paragraph; disallowed marks (`setMark`, `toggleMark`, the
`**bold**` shortcut) no-op. But a restricted editor still **renders** a
disallowed-but-registered block that arrives some other way — pasted, imported,
or synced from a collaborator on a looser schema. Paste and import are
normalized to the allow-list (a disallowed heading pastes as text; an image
drops); already-synced document content is not rewritten.

**Collaboration is safe.** The allow-list is an authoring constraint, never a
storage one — the CRDT log and the reducer ignore it entirely. Two peers with
_different_ allow-lists (say a locked title field and a full body) still
converge: the allow-list only governs what a peer can mint, never how it renders
the shared log. See [Collaboration](/docs/editor/collaboration).

```ts
const restricted = baseSchema.restrict({ blocks: ["heading1"], marks: [] });

// Query the allow-list to gate your own toolbar / slash menu chrome:
restricted.data.isBlockAllowed("bullet_list"); // false
restricted.data.isMarkAllowed("strong"); // false
restricted.data.hasBlock("bullet_list"); // true — still registered, still renders
```

<h2 id="window">View windows (title surfaces)</h2>

`restrict` decides which types an editor may author; a **view window** decides
which of a document's _blocks_ an editor renders and edits. Pass one as the
`window` option to `mountEditor` / `createEditor` / `useEditor`. Omit it (the
default) for a full-document editor.

A window's headline use is a **title editor**: a second, compact surface bound to
the _same_ `Doc` as the main body, scoped to just the title block. Because both
editors share one CRDT document, editing the title updates the body's heading
live — with no extra sync wiring (collaboration lives on the doc, not the
surface).

```ts
import {
  createDoc,
  createEditor,
  baseSchema,
  titleBlockWindow,
} from "@tasfer/editor";

const doc = createDoc({
  markdown: "# My page\n\nBody…",
  schema: baseSchema.data,
});

// Full body editor over the whole doc.
const body = createEditor({ element: bodyEl, doc });

// A title editor over the SAME doc — only the title block, only a heading1.
const title = createEditor({
  element: titleEl,
  doc,
  schema: baseSchema.restrict({ blocks: ["heading1"], marks: [] }),
  window: titleBlockWindow(),
});
```

- `titleBlockWindow()` scopes to the document's **title block** — the first
  text-bearing block (heading, paragraph, list, …), skipping leading images or
  dividers. It resolves the block structurally, so the window stays put even when
  the title is momentarily empty. `blockIdWindow(id)` scopes to one specific
  block; `titleBlockIndex(blocks)` returns the title block's index (or `-1`).
- These are **single-block** windows: the surface can never split (Enter is
  inert), merge into a neighbour (Backspace/Delete at the edges no-op), gain
  blocks from a paste (it flattens to inline text), and the caret cannot leave
  the window. It never mutates any block but its own.
- A window is a **view/authoring concern only** — never a storage one. The doc,
  reducer, and sync ignore it, so a windowed and a full editor on one doc always
  converge. Pair it with `restrict` to also constrain the single block's type
  (e.g. "only ever a `heading1`"); the block still _renders_ whatever it already
  is.

`ViewWindow` is the general shape (`{ select, singleBlock? }`) if you need a
custom rule — `select(blocks)` returns the set of original indices to show.
