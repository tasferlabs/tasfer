<h2 id="stack">Stack them</h2>

Providers compose. A typical setup persists locally *and* syncs over the network — the editor merges all of them into one converging document:

```ts
import { createIndexedDBProvider } from "@cypherkit/provider-indexeddb";
import { createRelayProvider } from "@cypherkit/provider-relay";

createIndexedDBProvider({ doc: editor.doc, name: "notes/today" });
const net = createRelayProvider({
  doc: editor.doc,
  room: "team-notes",
  relay: "wss://relay.cypher.md",
});
```

<h2 id="presence">Presence & remote cursors</h2>

Presence is ephemeral state that travels alongside the document but is never persisted — who is here, where their cursor is, what color to draw them. The protocol just routes whatever object you publish:

```ts
net.presence.set({ name: "Ada", color: "#1db984" });

// each entry is { peerId, state } — the state is what that peer published:
net.presence.on("change", (peers) => {
  avatars.render(peers.map((p) => p.state.name));
});
```

The editor engine itself has no concept of "peers" — it paints generic [decorations](/docs/editor/concepts#decorations) you feed it. To turn presence into remote carets and selections, bind the two together with the helper from `@cypherkit/provider-core/cursors`:

```ts
import { bindPresenceCursors } from "@cypherkit/provider-core/cursors";

// publishes your selection on every change, and renders every peer's cursor
// as a `presence:<peerId>` decoration layer. Returns an unsubscribe.
const stop = bindPresenceCursors(editor, net, {
  user: { peerId, name: "Ada", color: "#1db984" },
});
```

Syncing presence over your own transport instead of a provider? Call the pure mappers directly — `selectionToCursorPresence(editor.state.selection.range, user)` to publish, and `cursorPresenceToDecorations(peerId, state)` → ``editor.view.setDecorations(`presence:${peerId}`, …)`` to display.

<Callout kind="note" title="Offline is not a special case.">Disconnect, edit for an hour on a plane, reconnect — the CRDT merges your changes with everyone else's deterministically. There is no "resolve conflicts" dialog because there are no conflicts to resolve.</Callout>
