<h2 id="providers">Choosing a provider</h2>

<PropsTable cols={["Provider", "Transport", "Use it when"]} rows={[
  { name: "provider-relay", type: "WebSocket relay", desc: "Peers behind NATs/firewalls. The relay introduces peers and forwards their updates — it can read them, so run one you trust." },
  { name: "provider-webrtc", type: "Peer-to-peer", desc: "Direct browser-to-browser once introduced. Lowest latency; needs a small signaling step. Data channels are encrypted end to end." },
  { name: "provider-indexeddb", type: "Local disk", desc: "Not a network provider — persists the CRDT to IndexedDB so reloads are instant and offline-complete." },
  { name: "custom", type: "Anything", desc: "Implement the Transport interface (from @cypherkit/provider-core) and hand it to createProvider — sync over your own backend, BroadcastChannel, or a file watcher." },
]} />

<Callout kind="note" title="A relay sees your documents.">`provider-relay` does not encrypt: it base64-encodes frames so the relay never has to parse them, which is not the same thing. A relay operator can read and modify every document that passes through, and the peer id on each frame is self-asserted, so a peer in the room can forge frames attributed to another peer. Point it at a relay you control. `provider-webrtc` is the one that keeps the server out of the data path — once ICE completes, document bytes flow directly between peers over DTLS-encrypted DataChannels, and the signaling server only ever sees connection metadata.</Callout>

<h2 id="stack">Stack them</h2>

Providers compose. A typical setup persists locally *and* syncs over the network — the editor merges all of them into one converging document:

```ts
import { createIndexedDBProvider } from "@cypherkit/provider-indexeddb";
import { createRelayProvider } from "@cypherkit/provider-relay";

createIndexedDBProvider({ doc: editor.doc, name: "notes/today" });
const net = createRelayProvider({
  doc: editor.doc,
  room: "team-notes",
  relay: "wss://relay.cypher.md",
});
```

<h2 id="presence">Presence & remote cursors</h2>

Presence is ephemeral state that travels alongside the document but is never persisted — who is here, where their cursor is, what color to draw them. The protocol just routes whatever object you publish:

```ts
net.presence.set({ name: "Ada", color: "#1db984" });

// each entry is { peerId, state } — the state is what that peer published:
net.presence.on("change", (peers) => {
  avatars.render(peers.map((p) => p.state.name));
});
```

The editor engine itself has no concept of "peers" — it paints generic [decorations](/docs/editor/concepts#decorations) you feed it. To turn presence into remote carets and selections, bind the two together with the helper from `@cypherkit/provider-core/cursors`:

```ts
import { bindPresenceCursors } from "@cypherkit/provider-core/cursors";

// publishes your selection on every change, and renders every peer's cursor
// as a `presence:<peerId>` decoration layer. Returns an unsubscribe.
const stop = bindPresenceCursors(editor, net, {
  user: { peerId, name: "Ada", color: "#1db984" },
});
```

Syncing presence over your own transport instead of a provider? Call the pure mappers directly — `selectionToCursorPresence(editor.state.selection.range, user)` to publish, and `cursorPresenceToDecorations(peerId, state)` → ``editor.view.setDecorations(`presence:${peerId}`, …)`` to display.

<Callout kind="note" title="Offline is not a special case.">Disconnect, edit for an hour on a plane, reconnect — the CRDT merges your changes with everyone else's deterministically. There is no "resolve conflicts" dialog because there are no conflicts to resolve.</Callout>

<h2 id="schemas">Peers with different schemas</h2>

Two peers editing the same document do not need identical schemas. The CRDT log and the reducer are schema-agnostic about the [authoring allow-list](/docs/editor/api-schema#restrict): a peer records and materializes every well-formed op regardless of what its own schema permits it to _create_. So a peer whose schema is `restrict`ed to headings-only still renders a bullet list a collaborator produced — it just can't mint one locally. Convergence is unaffected because the allow-list never changes how the shared log is projected.

The same rule covers version skew: a peer running an older build that doesn't recognize a newer block type keeps the op in its log (so re-saving never drops a newer peer's content) and renders the block once its schema learns the type. Never enforce an allow-list — or drop an "unknown" op — inside a provider or the reducer; that is the one change that _would_ diverge replicas.
