<h2 id="change">change(fn)</h2>

يمرّر `editor.change` إلى رد النداء كائن `ChangeApi`. تضيف كل دالة تعديلًا دقيقًا واحدًا إلى الطابور وتعيد `this`، لذا يمكن تسلسل الاستدعاءات. يُعتمد رد النداء كله معًا — **مدخل تراجع واحد، وبث واحد، وحدث `change` واحد** — وتعيد `change` ما إذا كان شيء قد تغيّر فعلًا.

```ts
// one mark toggle
editor.change((c) => c.setMark("strong"));

// several edits, still ONE undo step
const changed = editor.change((c) =>
  c.setBlock({ type: "heading", level: 1 }).insertText("Untitled"),
);
```

إذا كان هدف دالة مفقودًا أو غير صالح، فلا تفعل شيئًا بصمت ولا تضيف شيئًا إلى الطابور؛ لذلك تعيد `change` التي تنتهي فارغة `false` ببساطة.

<Callout kind="note" title="change() هي المعاملة.">إن كنت قادمًا من ProseMirror أو CodeMirror: تمثل `change()` المعاملة في هذا المحرر. لا يوجد كائن `transaction` أو `transact` منفصل تبنيه ثم تطبّقه؛ فرد النداء *هو* حد المعاملة. يُعتمد كل تعديل في استدعاء `change()` واحد ذريًا: مدخل تراجع واحد، وبث واحد، وحدث `change` واحد يحمل [`ChangeTransaction`](/docs/editor/api-editor#events) واحدًا (`{ isRemote, ops }`) يضم دفعة عمليات CRDT كاملة. أما التعديلات في استدعاءات `change()` المنفصلة فهي معاملات منفصلة.</Callout>

<h2 id="targets">الأهداف: DocPoint وDocRange</h2>

تقبل كل دالة في `ChangeApi` **هدفًا اختياريًا** يُضبط افتراضيًا على المؤشر أو التحديد الحي، فيبقى الاستخدام الشائع موجزًا (`insertText("x")` و`setMark("strong")`)، بينما تستطيع الإضافة العمل عند موضع صريح ومستقر في CRDT. تعيد [واجهة القراءة](/docs/editor/api-editor) نوعي `DocPoint` و`DocRange` نفسيهما، فتقرأ الموضع ثم تمرّره مباشرة إلى التعديل.

```ts
// type at the caret (default target)
editor.change((c) => c.insertText("hi"));

    from: { block: id, offset: 0 },
    to: { block: id, offset: 5 },
  }),
);
```

لا تعرّف أنواع العقد البنيوية واجهات تخصيص. تستمد كلها من المخصّص المحمول أصلًا مع التغيير: تقرأ شيفرة المضيف `c.identities` داخل `change`، ويتلقى وجه `structured.create` في مواصفة العلامة المخصّص نفسه ضمن سياقه. ويرسل كلاهما تعديلات `content_edit` عادية:

```ts
editor.change((c) => {
  const shapeId = c.identities.nextId();
  c.editContent(blockId, contentId, {
    kind: "node_insert",
    node: {
      id: shapeId,
      type: "shape",
      placement: { parentId: contentId, slot: "children", orderKey: "a0" },
    },
  });
});
```

<Callout kind="note" title="العلامات القابلة للتبديل مقابل العلامات الحاملة للبيانات.">يبدّل `setMark(name)` بلا خيارات العلامات الأساسية القابلة للتبديل (`strong` و`emphasis` و`strike` و`code`) وأي علامة تبديل مخصصة عبر التحديد. تحمل `link` بيانات (URL) في `attrs`، لذا طبّقها بنطاق وسمات صريحة — `setMark("link", { attrs: { url }, range })` — وأزلها باستخدام `{ active: false, range }`. يؤدي تطبيق `math` على نطاق (عند تثبيت [`mathExtension`](/docs/editor/api-schema#math)) إلى **إنشاء** صيغة: يصبح النص المحدد مصدر LaTeX للمستند البنيوي الجديد، ويطوى النطاق إلى محرف مرساة واحد للعلامة.</Callout>

<Callout kind="note" title="محتوى الرياضيات البنيوي والإزاحات المسطحة.">نص المعادلة الرياضية المسطح فارغ، وشريحة المعادلة المضمّنة محرف مرساة واحد، لذلك لا تعنون نطاقات `{ block, offset }` *محتوى* الصيغة. تُرفض `insertText` و`deleteRange` الموجهتان إلى داخل كتلة رياضيات، ويتمدد النطاق الذي يقطع شريحة مضمّنة ليستبدلها كاملة. عدّل محتوى الصيغة عبر تعديلات `editContent` أو التحديد المتداخل التفاعلي.</Callout>

<Callout kind="note" title="المخططات المقيّدة.">
  في مخطط ضُيّق باستخدام [`restrict`](/docs/editor/api-schema#restrict)، يُتجاهل
  بهدوء أي `change` ينشئ نوعًا غير مسموح به: لا تفعل `setBlock` أو
  `insertBlock` ذات `type` المحظور شيئًا (مع استمرار تطبيق السمات الأخرى في
  الرقعة نفسها)، ولا تفعل `setMark` أو `toggleMark` لعلامة محظورة شيئًا. لا
  يُرمى خطأ؛ فالحقل المقيّد لا يستطيع إنشاء ما لا يسمح به. وفي المخطط الافتراضي
  غير المقيّد، تتصرف كل الأوامر تمامًا كما هو موثق.
</Callout>

<Callout kind="note" title="نمط كل كتلة.">تحمل كل كتلة مجموعة مفتوحة من تجاوزات العرض، وهي الطبقة الثالثة من تسلسل الأنماط (افتراض العقدة ◁ ‏`styles.blocks[type]` في السمة ◁ نمط الكتلة). اضبطها عبر مفتاح `style` المتداخل في `setBlock` أو `insertBlock` — مثل `c.setBlock({ style: { color: "#f00", fontSize: 20 } })` — أو ازرعها عند الإنشاء. كل خاصية سجل CRDT مستقل، فتندمج التعديلات المتزامنة لخصائص مختلفة بدل أن يطغى بعضها على بعض، وتمسح `setBlock({ style: { color: null } })` مفتاحًا واحدًا. تحترم كتل النص المضمنة مفاتيح `TextStyle` (`fontSize` و`fontWeight` و`color` و`lineHeight` و`paddingTop` و`paddingBottom`)؛ ويمكن للعقدة المخصصة قراءة ورسم أي مفتاح آخر تفهمه. يسقط تصدير Markdown النمط افتراضيًا لعدم وجود صياغة له؛ وعلى العقدة التي تريد الحفاظ عليه خلال التحويل أن ترمّزه في مُسلسل HTML أو Markdown الخاص بها.</Callout>

<h2 id="canchange">canChange(fn) — التجارب الجافة</h2>

تشغّل `canChange` رد النداء نفسه على نسخة مؤقتة من الحالة وتبلّغ عما إذا كان *سيغيّر* شيئًا، من دون لمس المستند. استخدمها لتفعيل واجهة المستخدم أو تعطيلها:

```ts
// is there anything to turn into a heading right now?
const enabled = editor.canChange((c) =>
  c.setBlock({ type: "heading", level: 1 }),
);
toolbarButton.disabled = !enabled;
```

<h2 id="history">التراجع والإعادة</h2>

السجل محلي ومدرك لـCRDT؛ فهو يحوّل بين مواضع الفهارس ومعرّفات CRDT ليبقى صحيحًا بعد دمج بعيد. ولا تدخل تعديلات النظير في سجل تراجعك.

```ts
editor.undo(); // returns whether it changed the document
editor.redo();
```

<h2 id="actions">الأفعال: المراقبة والتجاوز</h2>

فوق طبقة تعديل المستند توجد حافلة إرسال صغيرة، وهي وسيلة المضيف لربط سلوك المحرر من دون تشعيبه. يعرّف `action(name)` إشارة منمّطة تُعرّف **بالمرجع**؛ تُسجّل المعالجات بأولوية، وإما أن **تتجاوز** (تعيد `true`، فتتبناها وتوقف الانتشار) أو **تراقب** (`false` أو `void`). الحافلة خاصة بكل نسخة، لذلك يحتفظ محرران في صفحة واحدة بمستمعين مستقلين.

هناك ثلاثة أنواع متراتبة من الأفعال:

```ts
import { OPEN_LINK } from "@tasfer/editor";

// observe a built-in action — open links in the host's own router
const off = editor.registerAction(OPEN_LINK, ({ url }) => {
  router.push(url);
  return true; // claim it: skip the engine's default
});
```

أرسل فعلًا بنفسك باستخدام `editor.dispatch(action, payload)`. مع الفعل العادي، تكون القيمة المعادة ما إذا تبنّاه معالج؛ ومع `MutationAction` تكون ما إذا تغيّر المستند.

<Callout kind="note" title="مدخلان وآلية واحدة.">
  تمثل `change` الواجهة المريحة لتعديلاتك *أنت*، بينما حافلة الأفعال
  (`dispatch` و`registerAction`) للسلوك *المسمّى والقابل للربط* الذي يرسله
  المحرك أيضًا (قائمة الشرطة المائلة، ونقر الروابط، والإيماءات الأصلية). يشغّل
  `MutationAction` الذي ترسله افتراضه وكل مراقب داخل معاملة واحدة، بالآلية نفسها
  التي تستخدمها `change`: خطوة تراجع واحدة وبث واحد. (لا توجد دالة مساعدة `run`
  منفصلة؛ فتركيب عدة تعديلات ليس إلا رد نداء `change` واحدًا.)
</Callout>
